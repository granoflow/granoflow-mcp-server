# Legacy Task Plan Workflow

Use this reference only to resolve, execute, or audit a historical Task Plan.
It is not the Planning owner for new task work.

For every new task and material amendment, use
`task-work-document-workflow.md` with `task-work-document-template.md`.

A legacy Plan may execute only after the user explicitly instructs the host to
implement one unique, active, confirmed, content/hash-verified version. Legacy
`实施这个 Plan` remains a compatibility command. Draft, superseded, ambiguous,
missing, or unverified candidates fail closed.

Preserve the legacy Plan's nodes, dependencies, execution modes, Authorization
Matrix, verification, rollback, Delivery gate, summary markers, and App-owned
attachment identity. The task always belongs to the current individual user;
the Agent is an execution tool, not the task owner.

Before every node write, use current task state and reconcile conflicts. Create
and verify Task Delivery before the final required node or node-less
compatibility completion. Progress never rewrites the immutable legacy Plan.

Any material amendment enters the current Task Work Document Analysis and
Planning gates rather than creating another legacy Plan.
