# 76 final：快速插入任务最小打断契约

Status: superseded by v02 after grill-finalizer review

Date: 2026-07-13

## 用户目标

当用户在工作或学习过程中临时要求“插入任务”“加入任务”“把这个记成任务”或表达同等意图，且没有特别要求分析、规划、拆节点或选择归属时，Agent 应快速完成写入，不把任务捕获扩展成规划对话。

快速插入的目标是保存未来分析所需、事后难以恢复的上下文；完整分析、计划、节点拆分和任务回顾留到用户后续明确要求分析、开始、执行或完成任务时处理。

## 最终行为契约

### 默认快速路径

用户明确要求插入任务且没有附加处理要求时：

1. 从当前消息和紧邻对话中提取标题与最小描述。
2. 只使用已经明确可得的项目与活跃里程碑上下文判断归属。
3. 同时存在明确匹配的现有项目和该项目下的活跃里程碑时，直接绑定并写入。
4. 项目或活跃里程碑任一不明确时，省略 `projectId` 和 `milestoneId`，直接写入收集箱。
5. 用户的插入任务请求就是该次任务写入的确认；不再做 preview-confirm 往返。
6. 使用 structured create 真实写入，并做一次轻量回读确认任务实际存在且 description 未丢失。
7. 只向用户回复一句放置结果，不追加任务描述、建议、问题、ID、分析摘要或下一步。

### 唯一允许的两种默认回复

明确绑定项目和里程碑：

```text
任务已经放入「<项目名>」项目「<里程碑名>」里程碑。
```

进入收集箱：

```text
任务已收录到收集箱。
```

除非写入失败、Granoflow 不可连接或用户明确要求更多信息，默认回复不得增加第二句话。

## 最小描述契约

快速插入不生成完整分析，不机械填满长模板。description 只保留足以支持未来详细分析的要点：

```text
触发:
- 为什么现在记录这个任务，或当前观察到什么问题。

目标:
- 希望最终得到什么结果或发生什么变化。

线索:
- 当前已经明确的文件、页面、材料、错误、限制或相关对象。

完成信号:
- 当前已知的最直观完成判断；无法可靠判断时写“待分析”。
```

允许按任务规模省略没有信息的段落，但必须满足：

- 离开当前聊天后仍能理解为何存在这个任务；
- 至少保留目标或待解决问题；
- 不丢失用户已经给出的关键文件、材料、对象或限制；
- 不编造截止时间、验收标准、实现方案、项目归属或用户决策；
- 无法确定的内容明确留待分析，不为填满模板而猜测。

### 轻量任务类型适配

快速插入阶段可以内部判断任务更接近 `learning`、`software_development` 或 `general`，但分类不得触发额外问答、历史检索、节点拆分或完整 profile 展开。

- 学习任务优先保留：学习目标、材料、当前困难、期望获得的能力。
- 编程开发任务优先保留：当前行为、预期行为、仓库/文件/API 线索、验证提示。
- 通用任务优先保留：触发原因、目标、关键限制、完成信号。

任务类型暂时是 skill 内部判断，不新增持久化字段或公开 API schema。

## 明确省略的工作

默认快速插入路径不得执行：

- 分析文档或计划文档写作；
- 任务节点或子任务拆分；
- 任务执行、完成或状态推进；
- 任务回顾、知识卡片或经验卡片生成；
- 广泛历史检索、相似任务模糊搜索或完整去重审计；
- 新项目或新里程碑创建；
- 在多个弱匹配项目或里程碑之间要求用户选择；
- 未经用户表达的截止时间、提醒时间、优先级或标签推断；
- 为展示生成结果而向用户回显完整标题或 description。

允许使用一次有明确边界的精确重复检查；它不得让快速插入退化成长时间搜索。若无法快速确认重复，先完成收集箱写入，后续整理流程再处理。

## 归属规则

### 可直接绑定

只有以下条件全部成立时才直接绑定：

- 项目已经存在且与任务语义明确匹配；
- 里程碑已经存在、属于该项目且当前活跃；
- 归属可以从当前任务上下文、当前页面上下文或已解析的唯一候选直接得出；
- 不需要创建、恢复或重新开放项目/里程碑。

### 必须进入收集箱

出现以下任一情况时直接进入收集箱，不询问：

- 没有明确项目；
- 项目明确但没有明确的活跃里程碑；
- 多个项目或里程碑都可能匹配；
- 唯一候选里程碑已完成、归档、删除、废弃、过期或只是主题相似；
- 需要创建新项目或新里程碑；
- 需要依赖广泛搜索才能判断归属。

当前 API 中收集箱任务通过省略 `projectId` 与 `milestoneId` 表示；若运行中的 Granoflow 以后公开 app-owned inbox 字段或 capability，则使用 app-owned 表达方式。

## 与非快速路径的边界

以下用户表达应退出快速插入路径，路由到对应工作流：

- “插入并分析”“先分析这个任务”；
- “插入后给出计划”“拆成节点”；
- “放到某个项目/里程碑”，但指定结构不存在或含糊；
- “创建并开始执行”“完成这个任务”；
- “检查是否已有重复任务”；
- 用户明确要求预览、确认、完整描述或返回详细结果。

退出快速路径后，仍须遵守相应分析、执行、完成、卡片和安全确认契约。

## 写入与故障处理

默认调用顺序：

```text
提取最小任务
→ 确认是否已有明确项目 + 活跃里程碑上下文
→ granoflow_task_create_structured(dryRun=false)
→ 轻量回读
→ 一句话回复
```

仅在以下情况下打破一句话成功回复：

- Granoflow Local HTTP API 不可连接；
- structured create 写入失败；
- 回读发现任务不存在；
- description 或明确归属字段被丢弃且补写仍失败；
- 写入会包含超出用户请求的私人内容或秘密。

失败时应简短说明未成功写入及用户可采取的下一步，不得使用成功回复模板。

## 验收标准

- 用户仅要求插入任务时，不生成分析文档、计划文档、节点或卡片。
- 项目和活跃里程碑同时明确匹配时，任务真实绑定两者并完成回读。
- 项目或活跃里程碑任一不明确时，任务真实进入收集箱且不询问用户。
- 快速描述保留未来分析所需的触发、目标和已知线索，不编造缺失内容。
- 默认成功回复严格为允许的两种单句模板之一。
- 用户附加分析、规划、拆节点、执行或完成要求时，正确退出快速路径。
- structured write 或回读失败时不会误报成功。
- bundled workflow 与 shared task-copilot 不再保留相互冲突的归属确认规则。
- `npm run check` 通过。
- 仅修改 Markdown 的 shared skill 文件时，对改动文件运行项目可用的 Markdown lint；若实施包含 TypeScript 测试变更，则仍以 `npm run check` 为最终项目门禁。

## 回滚

若快速捕获导致错误归属或丢失关键上下文：

1. 回退 bundled/shared skill 中的快速捕获契约与相关 contract tests。
2. 恢复原有 discussed-requirement capture 行为。
3. 不需要数据库迁移或数据回滚；已经创建的任务保持原样，由用户或后续整理流程重新归属。

## 残余风险

- 过于保守的归属判断会让更多任务进入收集箱，但不会错误创建结构或打断用户；这是本 76 接受的折中。
- 一句话回复不会向用户展示标题，若 Agent 错误提取任务内容，只能在后续查看任务时发现；轻量回读只能证明写入一致，不能完全证明语义理解正确。
- 快速路径弱化模糊去重，可能产生少量重复任务；后续收集箱整理或显式去重流程负责处理。
- shared skill 位于另一个本地 skill 仓库，实施时必须分别验证该文件实际更新成功，不能只依赖当前仓库的 `git status`。
