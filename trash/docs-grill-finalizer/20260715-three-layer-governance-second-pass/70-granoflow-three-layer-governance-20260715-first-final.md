# 70 分析：Granoflow 通用 Agent 工作流的三层治理

Status: 70 analysis finalized after Grill Finalizer

Date: 2026-07-15

Mode: milestone

## 详细描述

本分析判断 Granoflow MCP 是否能够从“任务、复盘和长期工作记忆层”扩展为
开发前后端代码、Skill、MCP、Agent、文案写作、动漫创意和动画制作的通用工作
工具。目标不是让 MCP 自己实现所有专业能力，而是让 Granoflow 成为跨能力、
跨 Agent、跨产物的工作流治理与证据回写层。

用户真正期待的结果是：同一个工作系统能够理解不同类型的创作或工程任务，
帮助澄清问题、形成可靠计划、调用合适能力、保存中间与最终产物、验证真实结果，
并让下一次 Agent 能继续使用这些决策和经验。

当前判断：仅完善分析文档、计划文档和 Grill Finalizer，能够建立可靠的治理
闭环，但不能单独提供代码执行、图片/视频生成、浏览器、Figma、TTS、发布或
其他外部能力。因此需要把系统明确分为三层：治理层、能力层、编排层。

## 事实

- `granoflow-mcp-server` 当前是 TypeScript MCP server，主要通过 Local HTTP
  API 为 MCP 客户端提供任务、项目、里程碑、复盘、记忆和任务执行相关工具。
- 仓库规则要求 MCP 保持薄层：不直接访问 SQLite，不重做 Granoflow 业务逻辑，
  不把构建、截图、发布或其他应用编排逻辑塞进 MCP。
- 仓库已经有 `granoflow-agent-workflow`、`granoflow-daily-review`、
  `granoflow-review-card-draft`、`granoflow-first-run-import` 和
  `granoflow-gfmcp-runner` 等工作流 Skill。
- README 已经把 Task Analysis、Task Plan、Task Delivery、Task Review、
  Review Card 和历史上下文区分为不同阶段；第三方 Grill Skill 可以增强流程，
  但不能取代基础工作流。
- `grill-finalizer` 的责任是对具体 Markdown 或 `SKILL.md` 做问题展开、对抗式
  评审、决策确认、重写/演进和审计；它不是代码执行器、创作 Provider 或媒体
  产物管理器。
- 仓库已有 `npm run check`，覆盖 Prettier、ESLint、Python 静态检查、TypeScript
  构建和 Vitest；因此后续 MCP 代码变化具备基础质量门禁。

## 推断

### 1. 根问题不是缺少更多文档，而是缺少统一的能力边界

如果只继续增加 70/73 文档，系统会越来越擅长描述任务，却可能仍然不知道：

- 哪个工具能够执行当前工作；
- 谁拥有该工具的业务规则；
- 哪些输入可以发送到外部服务；
- 产物保存在哪里、如何复用、如何撤销；
- 什么证据才能证明工作真的完成。

因此，文档治理必须和能力注册、Provider 路由、执行适配器、产物管理及真实
结果验证绑定，但这些能力仍应保持在正确的责任层，而不是全部堆进 MCP。

### 2. 三层不是三个并列模块，而是三个不同的真相问题

| 层     | 它回答的问题                             | 主要真相                                        | 不应承担的责任                        |
| ------ | ---------------------------------------- | ----------------------------------------------- | ------------------------------------- |
| 治理层 | 这件事是否被正确理解、批准、规划和验证   | 分析、计划、决策、状态、授权、证据              | 执行专业工具或代替用户做审美/产品决策 |
| 能力层 | 哪些工具能完成哪一类工作，以及边界是什么 | Skill、MCP、Agent、Provider、工具契约、产物契约 | 决定整个任务方向或隐式扩大权限        |
| 编排层 | 当前任务如何选择能力并推动阶段流转       | 路由、上下文、执行、重试、等待、回写            | 重新实现各工具的业务逻辑或绕过确认门  |

三层之间必须通过显式的任务、计划、能力调用、产物和证据关系连接，不能只
依赖 Agent 的自然语言记忆。

## 三层责任判断

### 第一层：治理层

治理层是 Granoflow 最稳定、最应该优先拥有的核心。

它应统一管理：

- 需求捕获、问题分析、范围扩展和非目标；
- 分析文档、计划文档、决策日志和当前状态快照；
- `grill-finalizer` 的问题账本、自动确认项、用户确认项和延期项；
- 任务、项目、里程碑、计划节点、交付记录、复盘和长期记忆之间的关系；
- 用户授权、外部发送、付费、登录、发布、删除和不可逆操作的确认边界；
- “已生成”“已写入”“已运行”“已验证”“已发布”等不同完成状态；
- 失败、阻塞、取消、重试、回滚和部分完成的可见性。

治理层的核心 SSOT 是 Granoflow 的任务与工作记忆模型，以及最终确认的
分析/计划/决策/交付/复盘文档。`grill-finalizer` 是治理层的定稿机制，不是
治理层本身的全部数据模型。

治理层必须对非技术用户呈现可理解的结果：当前做什么、为什么这样做、需要
用户决定什么、已经产生什么、还有什么没有验证，而不是只返回工具调用日志。

### 第二层：能力层

能力层负责描述和提供专业动作。它可以包括：

- 前端、后端、测试、调试、重构、代码审查和发布能力；
- Skill 的创建、评审、演进、安装、版本和回归验证能力；
- MCP server 的设计、工具契约、测试、打包、目录注册和发布能力；
- Agent 的提示词、角色、上下文、权限、执行器和协作能力；
- 文案、品牌、SEO、产品说明、脚本和多语言写作能力；
- 角色设定、世界观、分镜、关键帧、风格锁定和动漫创意能力；
- 图片、视频、TTS、字幕、音频混音、屏幕合成和最终媒体检查能力；
- 浏览器、Figma、Git、文件系统、云端 API 和发布平台等外部 Provider。

每项能力至少需要有能力名、触发条件、输入/输出、权限、隐私边界、成本、
网络要求、可预览性、可撤销性、失败降级方式和验证方式。能力可以由本地
Skill、MCP 工具、Agent、脚本或外部服务提供；“能力”这个概念不能和某一种
实现方式混为一谈。

能力层的 SSOT 应是能力注册表及其各 Provider 的 Skill、工具契约、输入输出
Schema 和验证说明。Granoflow MCP 只应暴露稳定的发现、调用、状态和证据接口，
不应在自身内部复制每个专业工具的业务逻辑。

### 第三层：编排层

编排层把治理层的意图连接到能力层的实际执行。

它应能够：

- 从已确认的分析和计划中识别任务类型、阶段和所需能力；
- 根据本地可用性、权限、隐私、成本和证据要求选择 Provider；
- 在执行前展示需要确认的外部发送、付费、登录、发布或破坏性动作；
- 为长任务保存运行状态、输入版本、重试指纹、日志摘要和中间产物；
- 在工具不可用时给出明确阻塞或安全降级，不伪造完成；
- 将结果、产物、验证证据和失败原因写回治理层；
- 支持 Agent 继续执行，而不是每次从聊天上下文重新猜测状态。

编排层的 SSOT 是运行记录、能力调用记录、产物关联和验证证据；它不能把
“进程返回 0”直接等同于用户可见的完成，也不能越过 App、Provider 或用户的
授权边界。编排运行状态也不能替代 Granoflow 的任务状态：前者说明一次能力
调用进行到哪里，后者说明用户工作项处于什么生命周期阶段。

## 三层之间的稳定边界

### 允许的依赖方向

```text
治理层：意图 / 决策 / 计划 / 授权 / 验收要求
        ↓
编排层：能力发现 / 路由 / 执行 / 等待 / 回写
        ↓
能力层：代码、Skill、MCP、Agent、写作、图像、视频、音频、外部工具
        ↓
真实世界：仓库、App、API、文件、媒体产物、发布平台
        ↑
验证证据、产物状态、失败和授权结果逐层回写
```

治理层可以要求“需要一张角色一致性检查后的关键帧”，但不应规定具体图像
模型内部如何生成；能力层可以声明“支持角色参考图和多角度检查”，但不应
擅自把产物标记为最终交付；编排层可以选择合适 Provider，但不能把失败隐藏
成成功。

### 必须保持的概念区分

- 分析不是计划，计划不是执行，执行不是交付，交付不是复盘；
- Skill 是可复用工作流规则，MCP 是协议/工具暴露，Agent 是执行主体，Provider
  是具体能力来源；
- 产物存在不等于产物合格，工具成功不等于用户目标完成；
- 自动推荐不等于用户授权，历史决策不等于当前决策；
- 本地验证、公开可访问、已发布、第三方目录收录必须分别记录。
- 治理覆盖范围不等于当前 Provider 已实现范围；“支持某类工作”不等于所有
  该类工具已经可调用。

## 适用性迁移测试

### 前后端代码开发

治理层提供需求分析、计划、风险和验收证据；编排层选择代码 Agent、仓库工具、
测试和发布 Provider；能力层提供具体代码读写、测试、lint、构建和部署能力。
该场景验证了三层可以覆盖工程生命周期，但也证明 MCP 不能替代真实仓库和
项目质量门禁。

### Skill / MCP / Agent 开发

治理层记录能力目标、工具契约、权限和回归条件；编排层串联设计、实现、测试、
打包和发布；能力层由 Skill authoring、MCP SDK、Agent runtime 和 npm/目录
Provider 提供动作。该场景验证了“能力本身也可以作为被治理的工作对象”，
而不是只能作为黑盒插件。

### 文案写作

治理层保存受众、目的、品牌约束、事实来源、审阅意见和最终版本；编排层选择
写作、翻译、SEO 或事实核查能力；能力层生成草稿、改写和格式化产物。它验证
了治理层不能只面向代码字段，还要能表达审美、语气、事实和人工批准。

### 动漫创意与动画制作

治理层管理角色/世界观/风格锁定、镜头计划、版本、版权和最终验收；编排层
串联图像生成、视频生成、TTS、音频混合、字幕和屏幕合成；能力层保存模型、
提示词、参考图、关键帧和媒体处理 Provider。该场景验证了产物血缘、视觉一致
性和媒体级验收是通用系统必须补足的能力，不是普通 Task CRUD 能自然覆盖的。

迁移测试结论：三层边界可以横跨这五类工作，但能力注册、产物血缘、Provider
隐私/授权和用户可见验收必须作为设计预留，不能只靠现有任务字段承载。

第一阶段不应同时交付五类完整专业能力。应选择一个工程场景和一个创作场景
做最小纵向闭环，用真实产物和验证证据证明三层边界可复用；其余场景先完成
能力契约和 Provider 预留，不把战略覆盖范围表述成当前已交付能力。

## 范围分类

### 本轮纳入

- 将 Granoflow 的产品定位明确为“治理层 + 编排入口 + 工作记忆”，而不是
  所有专业能力的实现者；
- 确立治理层、能力层、编排层的责任边界、SSOT 和完成状态区分；
- 确立后续 73 计划必须覆盖能力注册、Provider 路由、产物/证据和真实验证；
- 保留 MCP 薄层原则，避免把专业业务逻辑复制到 MCP server。

### 设计预留

- Capability Registry / Provider Registry；
- 统一能力调用、运行状态、产物血缘和验证证据模型；
- 面向代码、Skill/MCP/Agent、文案、动漫和动画的能力 Profile；
- 本地优先、隐私、账号、付费、网络和发布授权策略；
- 适配现有 Granoflow App、Local HTTP API、MCP server 和共享 Skills 的边界。
- 第一阶段一个工程纵向闭环和一个创作纵向闭环的场景选择与验收边界。

### 非阻塞记录

- 具体要先实现哪一种专业能力；
- 是否采用某个具体模型、图片/视频 Provider 或第三方目录；
- 能力注册表最终存放在 Granoflow、Skill 仓库还是独立运行时；
- 是否为每一类创作工作建立独立项目模板。

停止扩展理由：继续增加具体模型、平台和创作模板不会改变当前根问题、责任
边界或验收判断；这些选择应留给后续 73 计划和对应 Provider 设计。

## 旧规则与当前意图冲突判断

- 旧规则：Granoflow MCP 是 Local HTTP API 的薄协议层，不实现代码分析、CI、
  SQLite、App 构建或媒体编排。
  当前意图：Granoflow 成为覆盖开发和创作的工具。
  判断：`auto_resolved_as_old_rule_still_valid`。新意图要求的是跨能力治理与
  编排，不要求 MCP 复制专业执行逻辑；薄层规则仍保护所有权和维护边界。

- 旧规则：Task Analysis、Task Plan、Execution、Delivery、Review 分阶段。
  当前意图：用 Grill Finalizer 提升分析和计划质量。
  判断：`auto_resolved_as_old_rule_still_valid`。Grill Finalizer 加强定稿，
  不应把分析直接变成计划或执行。

- 旧规则：第三方 Grill Skill 不是基础工作流的必要依赖。
  当前意图：集成 `grill-finalizer` 及辅助工具。
  判断：`auto_resolved_as_design_change`，但范围有限。可以将其作为增强型
  治理 Provider；基础分析、确认、计划和执行闭环仍须在没有它时可运行。
  后续长期文档应明确“增强依赖”和“基础能力”的区别。

## 结论

本需求的正确下一步是 `upgrade_73`，不是继续扩写本 70 文档，也不是立即为
所有创作类型各写一个孤立 Skill。

后续 73 计划应把本分析冻结为边界，重点规划：能力注册与 Provider 契约、三层
状态和数据关系、产物与证据模型、授权/隐私/失败恢复、MCP/App/Skill 的责任
分配，以及一个工程场景和一个创作场景的最小纵向闭环。其余三类场景只作为
能力 Profile 和迁移验证预留。计划必须保留“Granoflow 负责治理和
记忆、专业工具负责执行、编排层负责连接和验证”的核心判断。
